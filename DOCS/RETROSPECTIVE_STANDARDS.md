# 회고 작성 표준 가이드라인 (Retrospective Standards)

이 문서는 사용자와 AI가 회고를 작성할 때 따라야 할 **불변의 목차 구조**를 정의합니다.

---

## 1. 성공 회고 (Success Log)

> 문제를 맞혔을 때, 더 나은 코드를 위해 작성하는 양식입니다.

### [User 작성 영역]

1. **접근 방법 (Approach)**

   - 문제를 해결하기 위해 어떤 알고리즘이나 자료구조를 선택했나요?

   - 풀이의 핵심 로직을 한 줄로 요약해 보세요.

2. **복잡도 분석 (Complexity)**

   - 시간 복잡도: O(?)

   - 공간 복잡도: O(?)

### [AI 분석 영역] (AI-Generated)

3. **리팩토링 제안 (Refactoring)**

   - 현재 코드에서 가독성을 높이거나 더 효율적으로 줄일 수 있는 부분은 무엇인가요?

   - (Java/Kotlin Stream API 활용 등 구체적 제안)

4. **모범 답안 비교 (Best Practice)**

   - 다른 고수들은 이 문제를 어떻게 접근했나요? (내 풀이와의 차이점)

5. **심화 학습 키워드 (Deep Dive)**

   - 이 문제와 관련하여 추가로 공부하면 좋을 CS 개념이나 키워드

---

## 2. 실패 회고 (Failure Log)

> 오답이거나 풀지 못했을 때, 원인을 분석하고 배우기 위한 양식입니다.

### [User 작성 영역]

1. **실패 현상 (Symptom)**

   - 어떤 종류의 에러가 발생했나요? (시간 초과, 메모리 초과, 틀렸습니다, 런타임 에러)

   - 테스트 케이스 중 통과하지 못한 예시가 있나요?

2. **나의 접근 (My Attempt)**

   - 어떤 로직으로 풀려고 시도했나요?

### [AI 분석 영역] (AI-Generated)

3. **원인 분석 (Root Cause)**

   - 코드가 실패한 결정적인 논리적/문법적 원인은 무엇인가요?

4. **반례 제안 (Counter Example)**

   - 내 코드를 무너뜨리는 치명적인 입력값(Edge Case)은 무엇인가요?

5. **해결 가이드 (Guidance)**

   - 정답을 바로 알려주는 대신, 해결의 실마리가 되는 힌트 3단계

   - 수정해야 할 핵심 포인트
